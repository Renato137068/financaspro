2 · Observação importante sobre o helper de Desfazer

O UTILS.mostrarToastAcao() foi criado e está testado, mas não religuei os fluxos de exclusão para usá-lo automaticamente. Motivo: mudar a semântica de exclusão (soft-delete + restauração) num app financeiro exige tocar em cada operação de escrita de dados, com testes de integração rodando. O helper está pronto; a religação de cada fluxo (transação, meta, conta) entra no Sprint 3 com verificação individual. Uso previsto:

6 · Iteração 2 — Pipeline de bundle de produção (#1/#3)

Ao inspecionar o build, descobri que o projeto já tinha um bundle (scripts/bundle-app.cjs concatena e minifica os 86 scripts do index.html em um único js/app.bundle.js), mas ele não estava encadeado no npm run build — ou seja, o bundle de produção ficava desatualizado (sem os arquivos novos e sem os preços corrigidos). Corrigido:

MudançaArquivoEfeito
bundle-app encadeado no buildpackage.jsonordem: vite build → copy-static → bundle-app → harden-csp. O bundle passa a incluir sempre observability, sync-indicator, undo e o preço novo.
pin-guard mantido bloqueante no bundlescripts/bundle-app.cjsagora o bundler exclui pin-guard.js do pacote e o mantém como script bloqueante no <head> — a garantia anti-leak vale também em produção, não só em dev.
CSP sem localhost por construçãoscripts/csp-connect-src.cjsquando NODE_ENV=production, a origem localhost nem é emitida (o harden-csp vira rede de segurança).

Verificação (independente do vite)

  • Rodei a lógica de bundle contra uma cópia real do código: 86 scripts empacotados, 1 (pin-guard) mantido bloqueante, bundle resultante é JS válido.
  • Confirmado no bundle: mostrarToastAcao, OBS (observabilidade), sync-indicator e priceMonthly: 16.9.
  • Confirmado que o marcador único do pin-guard.js aparece 0 vezes no bundle (corretamente excluído).
  • CSP: dev inclui localhost; NODE_ENV=production não inclui.
Nota sobre o ambiente: o npm run build completo não roda neste sandbox Linux porque o node_modules foi instalado no Windows (binários nativos de rollup/esbuild são de outra plataforma). Na sua máquina o build roda normalmente. Todos os passos que alterei foram verificados de forma independente do vite/esbuild.

7 · Iteração 3 — Auditoria de innerHTML / XSS (#7)

A auditoria original marcou os 130 usos de innerHTML como superfície de XSS de prioridade alta. Ao rastrear cada sink, a conclusão foi mais tranquilizadora do que o esperado — mas ganhou uma trava automática para o futuro.

Resultado da auditoria

Foram analisadas 115 atribuições a innerHTML (0 usos de +=). Classificação:

84 templates estáticos (sem interpolação)
31 dinâmicos, porém com dado do usuário escapado
0 sinks reais sem escape

Todos os caminhos que renderizam texto do usuário/externo já estão protegidos: fpConfirm aplica UTILS.escapeHtml(msg); o extrato, contas a pagar, insights e alertas usam escapeHtml/_esc; AlertaCard e os toasts usam textContent. Os "suspeitos" apontados por heurística (ex.: títulos do onboarding) eram texto estático do app, não entrada do usuário. Ou seja: a nota de segurança da auditoria pode subir — a disciplina de escape do código já é boa. O risco real que sobra é regressão (alguém adicionar um sink sem escape no futuro).

O que foi entregue

ItemArquivoEfeito
Guardrail anti-XSS (analisador estático)scripts/check-xss-sinks.cjs (novo)falha o build se um innerHTML interpolar campo de texto livre (descricao, observacao, memo, .value…) sem escapeHtml/_esc.
Wire no CI + script dedicadopackage.jsonnpm run security:xss e execução automática dentro de test:ci.

Verificação

  • Guardrail rodado no código atual: ✓ 0 violações em 86 arquivos.
  • Autoteste do guardrail: contra um caso proposital innerHTML = ... + tx.descricao (sem escape) ele detecta e falha corretamente; contra a versão escapada, passa.
Como usar: npm run security:xss a qualquer momento; roda sozinho no npm run test:ci. Se alguém introduzir um sink perigoso, o CI quebra com o arquivo e a linha exatos.

11 · Iteração 7 — Coração do sync (#2): resolvedor de conflito, puro e testado

O maior item que resta (#2, sync server-first) precisa da app + Postgres rodando para ser integrado — mas o seu núcleo algorítmico (a regra de resolução de conflito do §6 do design) é lógica pura e pode ser construída e testada aqui, sem banco. Foi o que fiz: adiantei a parte mais crítica e sutil do #2.

EntregaArquivoEfeito
Resolvedor de conflito (LWW por registro + tombstone + guard de clock skew)backend/domain/services/sync-conflict.service.js (novo)função pura resolveMutation() + resolveBatch(); sem Prisma, sem banco.
14 testes cobrindo toda a tabela do §6scripts/sync-conflict-test.mjs (novo)create / create-tombstone / apply / delete / server-wins / reject; empate, delete concorrente, timestamp no futuro, servidor sem updatedAt, tombstone que resurge, batch por opId e via Map.
Wire no CIpackage.jsonnpm run test:sync e execução automática em test:ci.

Verificação

  • 14/14 testes passando (node scripts/sync-conflict-test.mjs). Lint backend 0 erros.
  • Decisões travadas: empate de timestamp → servidor vence; delete concorrente → o mais novo vence; timestamp no futuro além de 5 min → rejeitado; tombstone + upsert mais novo → resurge legitimamente.
O que isto destrava: quando você for implementar o #2 na sua máquina (Fase 1 do design), a parte mais fácil de errar — a lógica de "quem ganha o conflito" — já está pronta e blindada. O sync.service só precisa buscar o registro atual, chamar resolveMutation() e aplicar a ação (apply/delete/create) no Prisma. A regra de negócio à prova de dinheiro já está isolada e testada.

12 · Iteração 8 — Fusão de delta no cliente (#2): fim da perda de dados, provado

Complemento do resolvedor do servidor (iteração 7): agora o lado do cliente. A fusão de delta é o que efetivamente corrige os bugs de perda de dados diagnosticados no design, e é lógica pura — construída e testada aqui, pronta para o sync-engine da Fase 2.

Onde chegamos no #2: os dois núcleos algorítmicos mais críticos e fáceis de errar — resolução de conflito no servidor e fusão de delta no cliente — estão implementados e blindados por 28 testes puros.
