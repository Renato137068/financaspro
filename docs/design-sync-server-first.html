1. Diagnóstico do estado atual

O que existe hoje (confirmado no código):

  • Frontend localStorage-first. Dados vivem em fp-store-v2 / STORAGE_TRANSACOES etc. O sync (DADOS.sincronizarComApi()) faz GET /api/v1/state e o _mergeSnapshotLocal() substitui o array local inteiro via _storageSetRaw(...).
  • Escrita com push individual, mas falha silenciosa. _pushTransacaoApi, _deleteTransacaoApi, _pushContasApi… todos terminam em .catch(function(){ return transacao; }) — se estiver offline ou a API der 500, o push é perdido sem retry nem fila.
  • Backend já tem boa base. StateService.getSnapshot() monta o snapshot completo (até 1000 tx) com cache Redis de 30s. Modelos Prisma já têm updatedAt @updatedAt.

Os bugs concretos de perda de dados

#CenárioResultado hojeGravidade
D1Crio transação offline → push falha (engolido) → depois roda um sincronizarComApi() (pull)O pull faz full-replace e apaga a transação não sincronizada.CRÍTICO
D2Excluo item offline → DELETE falha → próximo pullSem tombstone, o item ressuscita vindo do servidor.CRÍTICO
D3Edito no aparelho A, depois abro o B com estado antigo e ele sincronizaFull-replace por snapshot = last-writer-wins no nível do snapshot inteiro, pode clobber.ALTO

D5IDs: cliente gera timestamp-random-counter (UTILS.gerarId), servidor gera uuidEspaços de ID divergentes; create não é idempotente (dedupe frágil via _apiId).ALTO
Raiz do problema: o cliente é a fonte da verdade e o servidor é um espelho best-effort. Num app de dinheiro isso é invertido. A correção é tornar o servidor autoritativo e o local um cache que nunca descarta escrita não confirmada.

2. Objetivo e princípios

  • Servidor é a fonte da verdade. O local é cache offline.
  • Nenhuma escrita se perde. Toda mutação entra numa fila durável (outbox) e só sai após ACK do servidor.
  • Pull vira merge incremental por registro (delta por updatedAt + tombstones), nunca full-replace destrutivo.
  • Conflito é determinístico (LWW por updatedAt, regra única, sem merge parcial de campos de valor).
  • Idempotência ponta a ponta (mesma mutação aplicada 2× = 1 efeito).
  • Não-quebra: tudo atrás de feature flag, com o caminho antigo como fallback até a validação.

3. Arquitetura alvo

UI (módulos) Backend (autoritativo) | grava | v v DADOS.salvar*() --enfileira--> OUTBOX (localStorage: fp-outbox) POST /api/v1/sync | | flush (online/foreground/timer) | | lê +----------------------------------> | aplica LWW + tombstone v | grava Postgres CACHE local (fp-store-v2) <--merge incremental-- GET /api/v1/sync?since=cursor ^ (delta: upserts + deletes + novo cursor) | render telas

Três peças novas no cliente, uma no servidor:

  • novo js/core/sync-engine.js — outbox durável, flush com backoff, aplicação do delta no cache, cursor.
  • alterado js/core/dados.js — salvar*/excluir* enfileiram na outbox em vez de .catch() silencioso; pull vira merge incremental.
  • novo backend/domain/services/sync.service.js + rota /api/v1/sync (pull delta + push batch).
  • alterado Prisma: deletedAt (tombstone) e updatedAt exposto nos selects.

4. Mudanças no modelo (Prisma)

Adicionar soft-delete (tombstone) e garantir updatedAt em todas as entidades sincronizáveis: Transaction, Account, Budget, RecurringTransaction.

// prisma/schema.prisma — por entidade sincronizável
model Transaction {
  // ... campos atuais ...
  updatedAt   DateTime  @updatedAt        // já existe — passar a EXPOR
  deletedAt   DateTime?                     // NOVO: tombstone (null = ativo)
  @@index([userId, updatedAt])            // NOVO: varredura de delta rápida
}
Idempotência de ID (resolve D5): passar o cliente a gerar UUID v4 e o servidor a fazer upsert por id. Assim, reenviar uma criação não duplica. Migração: manter _apiId como ponte durante a transição; novos registros já nascem com UUID canônico.

Migração de banco (não destrutiva; colunas nulas):

# adiciona deletedAt e índices, sem downtime
npx prisma migrate dev --name add_sync_tombstones_and_updatedat_index

Nas queries de leitura, trocar delete físico por update { deletedAt: now() } e filtrar deletedAt: null no snapshot normal — mas incluir os tombstones no endpoint de delta.

5. Contrato de sync (API)

Pull incremental — GET /api/v1/sync?since=<cursor>

Retorna tudo que mudou desde o cursor (upserts + deletes), com um novo cursor. Cursor = timestamp do servidor (autoridade única de relógio) mais um desempate por id.

// resposta
{
  "cursor": "2026-07-09T14:03:11.482Z|9f2c...",   // opaco para o cliente
  "hasMore": false,
  "changes": {
    "transactions": [
      { "id":"...", "type":"expense", "amount":"42.90", ..., "updatedAt":"...", "deletedAt":null },
      { "id":"...", "deletedAt":"2026-07-09T13:59:00Z" }   // tombstone
    ],
    "accounts": [ ... ], "budgets": [ ... ], "recurringTransactions": [ ... ]
  }
}

Implementação: where: { userId, updatedAt: { gt: sinceTs } } ordenado por updatedAt, id, paginado (take: 500, hasMore). Primeira sync = since vazio → snapshot completo paginado (reaproveita a lógica do StateService).

Push batch (flush da outbox) — POST /api/v1/sync

// requisição
{
  "mutations": [
    { "opId":"uuid-idem", "entity":"transaction", "op":"upsert",
      "id":"uuid", "clientUpdatedAt":"2026-07-09T13:50:00Z", "payload": { ... } },
    { "opId":"uuid-idem", "entity":"transaction", "op":"delete",
      "id":"uuid", "clientUpdatedAt":"2026-07-09T13:52:00Z" }
  ]
}
// resposta
{
  "results": [
    { "opId":"uuid-idem", "status":"applied", "record": { ...authoritative... } },
    { "opId":"uuid-idem", "status":"stale",   "record": { ...server-wins... } }
  ],
  "cursor": "..."   // cliente pode encadear um pull a partir daqui
}
  • opId = chave de idempotência (dedupe de retries; guardar num set/janela ou tabela SyncOp).
  • Cada mutação é validada com Zod (já usado no projeto) antes de aplicar.
  • Status por item: applied · stale (servidor mais novo, devolve autoritativo) · invalid.

6. Resolução de conflito

Regra única, simples e à prova de dinheiro: Last-Writer-Wins por registro inteiro, comparando clientUpdatedAt (da mutação) com o updatedAt armazenado.

SituaçãoDecisão
Registro não existe no servidorcria (upsert)
clientUpdatedAt > servidor.updatedAtaplica a mutação do cliente
clientUpdatedAt ≤ servidor.updatedAtservidor vence; devolve registro autoritativo (status:"stale"), cliente sobrescreve o cache
delete vs update concorrenteso mais recente por timestamp vence; delete cria tombstone
Por que LWW de registro inteiro (e não merge de campos): mesclar campos de uma transação (valor, categoria, data) pode gerar um registro financeiro incoerente que nunca existiu. Substituir o registro inteiro é previsível e auditável. Para um app pessoal multi-dispositivo, LWW é o padrão correto; conflitos reais são raros (mesmo usuário).

Opcional (fase posterior): registrar conflitos num AuditLog (já existe no schema) e, se o delta perdido for material, notificar o usuário ("esta alteração foi substituída por uma mais recente").

7. Fila offline (outbox)

Núcleo do "nunca perder escrita". Cada mutação local é enfileirada de forma durável antes/junto de gravar o cache.

// js/core/sync-engine.js (ES5, estilo do projeto) — esqueleto
var SYNC_ENGINE = {
  KEY: 'fp-outbox',
  _fila: null,
  _lerFila: function(){ try { return JSON.parse(localStorage.getItem(this.KEY)||'[]'); } catch(e){ return []; } },
  _salvarFila: function(f){ localStorage.setItem(this.KEY, JSON.stringify(f)); },

  enfileirar: function(entity, op, id, payload){
    var f = this._lerFila();
    f.push({ opId: UTILS.gerarUuid(), entity: entity, op: op, id: id,
             clientUpdatedAt: new Date().toISOString(), payload: payload, tentativas: 0 });
    this._salvarFila(f);
    APP_STORE.sync.setPending(true);
    this.flush();                 // tenta já; se offline, fica na fila
  },

  flush: function(){
    if (!navigator.onLine || !DADOS._apiAtiva()) return Promise.resolve();
    var f = this._lerFila(); if (!f.length) return Promise.resolve();
    return DADOS._apiFetch('/api/v1/sync', { method:'POST',
        body: JSON.stringify({ mutations: f.slice(0, 100) }) })
      .then(function(res){
        SYNC_ENGINE._aplicarResultados(res.results);   // remove aplicados/stale da fila
        SYNC_ENGINE._puxarDelta(res.cursor);            // encadeia pull
      })
      .catch(function(){ /* mantém na fila; backoff agenda novo flush */
        SYNC_ENGINE._agendarRetry();
      });
  }
  // + _aplicarResultados, _puxarDelta(cursor), _agendarRetry (backoff exp.), gatilhos
};

Gatilhos de flush

  • evento online, foreground do app (visibilitychange), após login, timer periódico (ex.: 30s) e imediatamente após cada enfileirar.

Merge do pull (resolve D1/D2/D3)

  • O delta é aplicado registro a registro no cache. Antes de sobrescrever, verifica a outbox: se o id tem mutação pendente mais nova, o local é preservado (a outbox reenvia).
  • Tombstone (deletedAt) remove o item do cache — sem ressurreição.

Ponto-chave: substituir o .catch() silencioso

Em dados.js, os _push*Api(...).catch(()=>x) deixam de existir; os salvar*/excluir* passam a chamar SYNC_ENGINE.enfileirar(...). O cache local é atualizado na hora (UX otimista); a durabilidade fica com a outbox.

8. Migração faseada (feature flag)

Tudo atrás de config.syncV2Enabled (sistema de config já existente via DADOS.getConfig()), com o caminho antigo (/state full-replace) intacto como fallback.

0
Schema (não-quebra). Migration adiciona deletedAt + índice [userId, updatedAt]; expõe updatedAt nos selects. Cliente antigo ignora os campos extras. deploy isolado
1
Endpoints novos. GET/POST /api/v1/sync + sync.service.js, coexistindo com /state. Cobertos por testes de integração (supertest). Nada no cliente ainda.
2
Cliente atrás de flag. sync-engine.js + outbox + merge incremental, ativado só com syncV2Enabled. Dogfood interno; comparar com o caminho antigo.
3
Rollout gradual + soft-delete real. Trocar deletes físicos por tombstone. Habilitar para % de usuários; monitorar métricas (§12).
4
Depreciar o full-replace. Após validação, /state vira apenas "primeira carga" e o merge destrutivo é removido.

11. Riscos e mitigações

RiscoMitigação
Clock skew do cliente (LWW por timestamp)O cursor usa sempre o relógio do servidor. clientUpdatedAt só desempata a mutação do próprio usuário; para segurança, o servidor pode rejeitar timestamps futuros (> now + margem).
Espaço de ID divergente (D5)UUID v4 no cliente + upsert por id; ponte _apiId durante a transição.
Precisão monetária (Decimal(15,2))Enviar valor como string/inteiro de centavos; nunca float no fio.
Primeira sync grandePull paginado (take:500, hasMore) + skeleton; não bloquear a UI.
PII na outbox (localStorage)Integra com a cripto at-rest (#8): outbox cifrada quando a passphrase/PIN estiver ativa.
Regressão durante rolloutFeature flag + caminho antigo como fallback + testes de regressão D1/D2 no CI.

12. Métricas de sucesso

  • 0 perda de dados nos testes de regressão D1–D3 (gate de CI).
  • Outbox drain: % de mutações confirmadas < 60s após voltar a ficar online (meta > 99%).
  • Taxa de conflito (stale/total) monitorada — deve ser baixa; picos indicam bug de merge.
  • Tamanho médio da outbox por usuário (deve tender a 0; crescimento = flush travado).
  • Instrumentar via o OBS.track() já adicionado na iteração 1.
Resumo executável: começar pela Fase 0+1 (schema + endpoints, invisíveis ao usuário e 100% testáveis no backend com supertest), depois a Fase 2 no cliente atrás de flag. Cada fase é um PR pequeno, verificável e reversível. O gate que importa: os testes de regressão D1/D2 rodando verdes no CI antes de habilitar para qualquer usuário real.
