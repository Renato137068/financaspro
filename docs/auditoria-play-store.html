2. Obrigatório no Console (mesmo em teste fechado)

O Google exige a seção "Conteúdo do app" completa antes de publicar em qualquer trilha, inclusive teste fechado. São formulários, rápidos:

  • Data safety: declare que coleta dados financeiros (transações; e dados bancários se usar Open Finance de verdade), criptografados em trânsito, e que o usuário pode pedir exclusão. Seja preciso — o Google cruza com as permissões.
  • Classificação etária (IARC): questionário curto → provável "Livre/Everyone", mas precisa ser respondido.
  • Público-alvo: declare 18+ (evita a política de Famílias, comum em apps financeiros).
  • Anúncios: declare "não contém anúncios" (bate com sua proposta "sem anúncios").
  • Recursos financeiros: declare honestamente — "gerenciador de finanças pessoais". Se conectar contas via Open Finance, marque o que se aplica; empréstimo/pagamento/cripto = "não".
  • Ficha: título, descrição curta (≤80) e completa, categoria Finanças, ícone 512, feature graphic e ≥2 screenshots — você já tem os assets.

3. A decisão de backend (o item nº 1)

No app empacotado, CONFIG.API_BASE_URL é '' e não é injetado no build → _apiBaseUrl() cai para window.location.origin = https://localhost (esquema interno do Capacitor). Resultado: as chamadas de API não chegam a nenhum servidor. Duas saídas:

Opção A — Hospedar o backend (login/sync funcionam)

  • Suba o backend Express+Postgres num host (Railway, Render, Fly, VPS) com HTTPS, ex.: https://api.financaspro.com.br.
  • Defina CONFIG.API_BASE_URL para essa URL antes de buildar, e builde com PUBLIC_API_URL=https://api.financaspro.com.br (para a CSP liberar o domínio no connect-src).
  • Rode as migrations (incl. o updatedAt que já exponho) e o seed de planos.

Opção B — Piloto em modo local (mais simples para hoje)

  • Shipar sem backend: o app já tem modo local com localStorage. Esconda/adie o login para o tester só usar offline.
  • Prós: zero dependência de servidor, nada quebra. Contras: sem contas, sem sync entre aparelhos, sem billing.
  • Cuidado que já sinalizei: em modo local o dado vive só no aparelho — avise os testers que é piloto e pode resetar.
Recomendação para amanhã: se o backend não estiver deployado e testado, vá de Opção B para o piloto — é mais seguro do que subir um app cujo login falha. Migre para a Opção A quando o backend estiver de pé (e aí o #2 de sync entra).

4. Pode esperar (é piloto / teste fechado)

  • Os "12 testers por 14 dias": isso é requisito para graduar para produção em conta de desenvolvedor pessoal — não para rodar o teste fechado. Para o piloto, basta a lista de testers.
  • Polimento da ficha, tradução, vídeo, screenshots extras.
  • android:allowBackup="false" + regras de extração de dados — recomendado para app financeiro (evita dado sensível em backup na nuvem), mas não bloqueia o piloto.
  • Integração completa do sync (#2), cripto derivada do PIN (#8) — evoluções pós-piloto.

5. Checklist da véspera

  • Decidir backend: Opção A (hospedar + setar API_BASE_URL) ou Opção B (modo local).
  • Hospedar privacidade.html num URL público e anotar o link.
  • Confirmar a keystore local + senha; gerar o .aab assinado.
  • Testar o .aab num aparelho real (instalar via internal testing) — abrir, criar transação, ver o dashboard, fechar e reabrir.
  • No Console: Data safety, classificação etária, público-alvo (18+), anúncios (não), recursos financeiros, ficha (Finanças) + política de privacidade.
  • Criar a trilha de teste fechado e adicionar os e-mails dos testers.
  • Subir o .aab, mandar o link de opt-in para os testers.
